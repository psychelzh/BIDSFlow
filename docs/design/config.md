# `bidsflow.toml`

## 1. Why BIDSFlow needs a config file

BIDSFlow needs a project-level config file because the CLI should not
force users to repeat the same workspace assumptions in every command.

At a minimum, BIDSFlow needs a stable place to record:

- what directory is considered the project workspace
- where source data live
- where the curated raw BIDS dataset lives
- where derivatives, work files, logs, and internal state live

This gives BIDSFlow a durable project contract that can be inspected,
edited, versioned, and explained.

`bidsflow.toml` should therefore act as:

- the editable project scaffold created by `bidsflow init`
- the canonical home for BIDSFlow-owned defaults
- a stable place to document why those defaults were chosen

When future config sections are added, this document should be extended
rather than split into disconnected notes.

## 2. Current supported default config

As of April 2, 2026, the scaffold produced by `bidsflow init` supports
the following parameters.

When `--name` is omitted, the generated `name` defaults to the target
directory name. In the example below, `<target-directory-name>` stands
for that value.

```toml
# Generated by `bidsflow init`.
# Review before first use:
# - adjust [project].name if you want a clearer project label
# - adjust [paths] if your project layout differs from this scaffold

[project]
name = "<target-directory-name>"
root = "."

[paths]
source_root = "sourcedata"
raw_bids_root = "sourcedata/raw"
derivatives_root = "derivatives"
work_root = "work"
logs_root = "logs"
state_root = "state"
```

## 3. Section-by-section reference

### 3.1 `[project]`

#### `name`

Meaning:

- a human-readable label for the BIDSFlow project

Current default:

- target directory name, unless overridden by `--name`

Why this default:

- the scaffold should create a usable label without asking extra
  questions
- the target directory name is the least surprising local default
- users can still replace it immediately with `--name` or by editing the
  config

Source notes:

- This is a BIDSFlow usability choice rather than a BIDS-standardized
  field.

#### `root`

Meaning:

- the root directory of the BIDSFlow project workspace

Current default:

- `.`

Why this default:

- the generated config should be portable when moved together with the
  project directory
- relative roots keep the scaffold easy to read and easy to relocate
- this makes `bidsflow.toml` self-contained inside the workspace it
  describes

Source notes:

- This is a BIDSFlow design choice that keeps project layout explicit.

### 3.2 `[paths]`

#### `source_root`

Meaning:

- the root directory for source data that are not yet the curated raw
  BIDS dataset

Current default:

- `sourcedata`

Why this default:

- `sourcedata` is the official BIDS name for source data
- this keeps source inputs separate from curated raw BIDS data and from
  derivatives

Source notes:

- BIDS common principles, source data in `sourcedata`:
  [https://bids-specification.readthedocs.io/en/latest/common-principles.html](https://bids-specification.readthedocs.io/en/latest/common-principles.html)

#### `raw_bids_root`

Meaning:

- the root directory of the curated raw BIDS dataset that BIDS Apps will
  typically consume

Current default:

- `sourcedata/raw`

Why this default:

- the official BIDS study-dataset example shows a larger workspace with
  `sourcedata/dicoms/`, `sourcedata/raw/`, and `derivatives/`
- that layout matches BIDSFlow well if `project.root` is treated as a
  project workspace rather than as the raw BIDS dataset root itself
- it cleanly separates vendor or source material from the curated raw
  BIDS dataset

Caveat:

- if the project root itself is meant to be the raw BIDS dataset root,
  then `.` is also a reasonable configuration choice

Current recommendation:

- keep `sourcedata/raw` as the scaffold default for now
- document clearly that it is a workspace-oriented default, not a BIDS
  universal

Source notes:

- BIDS common principles, study dataset example:
  [https://bids-specification.readthedocs.io/en/latest/common-principles.html](https://bids-specification.readthedocs.io/en/latest/common-principles.html)
- HeuDiConv reproin documentation, example curated output layout:
  [https://heudiconv.readthedocs.io/en/latest/reproin.html](https://heudiconv.readthedocs.io/en/latest/reproin.html)

#### `derivatives_root`

Meaning:

- the root directory under which derived datasets and pipeline outputs
  should be written

Current default:

- `derivatives`

Why this default:

- `derivatives/` is the official BIDS location for derived datasets
- BIDS derivatives are expected to live under pipeline-specific
  subdirectories within that tree
- this matches the common BIDS App split between input dataset root and
  output derivatives root

Source notes:

- BIDS common principles, storage of derived datasets:
  [https://bids-specification.readthedocs.io/en/latest/common-principles.html](https://bids-specification.readthedocs.io/en/latest/common-principles.html)
- BIDS dataset description, derivative dataset metadata:
  [https://bids-specification.readthedocs.io/en/stable/modality-agnostic-files/dataset-description.html](https://bids-specification.readthedocs.io/en/stable/modality-agnostic-files/dataset-description.html)
- XCP-D usage:
  [https://xcp-d.readthedocs.io/en/stable/usage.html](https://xcp-d.readthedocs.io/en/stable/usage.html)

#### `work_root`

Meaning:

- the root directory for transient working files

Current default:

- `work`

Why this default:

- BIDS does not standardize working directories
- major BIDS Apps explicitly separate working directories from both
  inputs and outputs
- a top-level `work/` is easy to understand and keeps transient files
  out of raw and derivative datasets

Caveat:

- cluster users may later prefer a scratch-oriented location instead

Source notes:

- fMRIPrep usage:
  [https://fmriprep.org/en/stable/usage.html](https://fmriprep.org/en/stable/usage.html)
- MRIQC usage:
  [https://mriqc.readthedocs.io/en/latest/usage.html](https://mriqc.readthedocs.io/en/latest/usage.html)
- QSIPrep usage:
  [https://qsiprep.readthedocs.io/en/latest/usage.html](https://qsiprep.readthedocs.io/en/latest/usage.html)
- XCP-D usage:
  [https://xcp-d.readthedocs.io/en/stable/usage.html](https://xcp-d.readthedocs.io/en/stable/usage.html)

#### `logs_root`

Meaning:

- the root directory for BIDSFlow-owned orchestration logs

Current default:

- `logs`

Why this default:

- BIDS does not define a top-level orchestration log root
- BIDSFlow may still benefit from a separate location for scheduler and
  orchestration logs
- `logs/` is easy to find, easy to archive, and does not pollute raw or
  derivative trees

Source notes:

- BIDS common principles, derivatives may include a `logs/` subtree:
  [https://bids-specification.readthedocs.io/en/latest/common-principles.html](https://bids-specification.readthedocs.io/en/latest/common-principles.html)
- This top-level choice is still a BIDSFlow design decision.

#### `state_root`

Meaning:

- the root directory for BIDSFlow internal bookkeeping such as run
  state, resumability metadata, and handoff checks

Current default:

- `state`

Why this default:

- BIDS does not define orchestration state storage
- BIDSFlow should keep internal state outside raw inputs and outside
  derivative outputs
- `state/` clearly communicates "internal bookkeeping" instead of data
  products

Source notes:

- This is a BIDSFlow design choice rather than a BIDS-standardized path.
- Supporting rationale:
  [handoff-contract.md](handoff-contract.md)

## 4. Default summary

Defaults with strong support from existing ecosystem conventions:

- `source_root = "sourcedata"`
- `derivatives_root = "derivatives"`

Defaults that are reasonable but depend on BIDSFlow's chosen workspace model:

- `raw_bids_root = "sourcedata/raw"`
- `work_root = "work"`

Defaults that are primarily BIDSFlow-owned orchestration choices:

- `logs_root = "logs"`
- `state_root = "state"`

## 5. Future research slots

This section is intentionally left open for later additions.

Candidate follow-up topics:

- whether BIDSFlow should support an alternate scaffold preset where the
  project root is itself the raw BIDS dataset root
- whether cluster-oriented projects should default `work_root` to a
  scratch-oriented placeholder instead of `work/`
- whether `logs_root` should remain top-level or move under `state/`
- whether future config sections should distinguish raw BIDS inputs from
  multiple derivative inputs more explicitly
